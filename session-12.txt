Java
=====
Build tool -> Ant, Maven, Gradle, etc.
nowadays maven and gradle are used
file extension -- .java

JDK -> Java development kit
JRE -> Java Runtime environment

while developing application we need JDK
while running JRE is enough
JDK & JRE are just softwares
JDK = JRE + development utilities

we need to compile the source code then we get byte code

Source code (high level code) -> Compile -> Byte code (intermediate code) -> 0 and 1(low level code)
node js execution = compile + running at a time
java = first compilation and then running directly
-- by this application runs a bit faster compared to nodejs application

Build tool = Maven
output format after compilation -> .ear, .war, .jar
.ear -- enterprise archive file
.war -- web archive file
.jar -- java archive file
.jar has inbuilt lightweight server

Old Enterprise applications -> IBM Websphere, Oracle weblogic, JBOSS EAP, etc.. also known as middleware 
===========================
they install  IBM Websphere, Oracle weblogic, JBOSS EAP (heavy weight servers) and then they download application 
used in banking applications
enterprise archive file = frontend+backend+libraries -> .ear -- roboshop.ear

disadvantages
1. tight couple
2. if there is a small problem makes both frontend and backend down
3. a small change is also costly. for any small change, end to end testing should be done again
4. everytime frontend components(Jquery, Angular JS, etc.) should be downloaded from server
by that time and effort gets wasted

here they use physical servers, once u order physical server it takes atleast one month to deliver
if the application is big , it contains huge infra so cost increases, so scaling of application becomes difficult

Monolithic -> Apache Tomcat, JBOSS Application Server, etc.(lightweight)
=============
.war file
seperate frontend and seperate backend

advantages
1. easy maintanence
2. data only will be served by backend through API 
3. loose coupling

problems
=========
1. problem in catalogue makes entire backend down
2. entire application should be developed in same programming language
3. tight coupling between backend components

here they used virtualization -- logical division of computer hardware into different virtual servers, in virtual servers we can run install any os
if problem comes in monolithic it takes atleast 30mins to find where the problem is

Microservices -> inbuilt servers
=============
catalogue, user, cart, etc..

1. each component is independent
2. diff programming languages can be used
3. problem in one component cannot disturb other components

these are very small there is no need to run a vm
when application size decreases huge servers are not needed
functionality is same we divided the components for easy maintenance

nano services -- signup as one application & signin as other application
application size becomes low
