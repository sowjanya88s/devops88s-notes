inode symlink hardlink
======================

inode = stores metadata of the file/directory

file metadata -- extra info about file

Linux stores metadata of the file in inode i.e
file type --> file/directory/link
owner & group details
Permissions (rwx)
size of file
file creation & updation  details etc
actual memory block address

command to see info about inode number of all files in current path--

to see the info in inode --
stat Linux.txt
in output we can see size , blocks file type , links , modified & change & birth details

enter some text in file
stat Linux.txt
now info changes and inode number also changes

inode does not store actual file content


symlink == shortlink == softlink
1. it is like short cut to the file/folder
2. inode number is different for symlink
3. if actual file is removed, then symlink will break
4. uses for backward compatability, switching versions and rollback is easy
5. You can create symlink for folders and files also
6. You can create symlink across different filesystem 
filesystems can be -- xfs , ext3 , ext4

now create one directory here
mkdir softlinks
command to create softlink of a file -- 
syntax -- ln -s or ln <original-file-name> <link-file-name>
ln -s ../Linux.txt soft-link
this will create softlink of file Linux.txt in links folder with name soft-link
permissions starting with l -- means softlink  && soft-link file points to original file Linux.txt
now give command
stat soft-link
inode numbers of original file and softlink file are different

Hardlink
=========
1. You can't create hardlink in another filesystem
2. Hardlink is same as original file but can be different name
3. hardlink inode numbers are same
4. they points to same memory location, so memory is not wasted
5. you can't create hardlink to folders
6. hardlinks are used for backup.
7. when all links are deleted, then only file will be deleted.
if original file is deleted link file points to same memory block so it shows same data as original file. so hardlink works as backup

note: files in different file systems can contain same inode number. it treats each mount point as different file system
create empty file
touch aws.txt
creation of hardlink
ln aws.txt aws-1.txt
check inode numbers
ls -li
inode numbers will be same

version control system is of two types
1. centralized
2. distributed  

centralised vs distributed
===========================
in olden days, SVN is centralized version control system , code versions only available in SVN , our laptops doesn't contain different versions of code , our laptop contains only present version of code. so if SVN crashes , we can't get previous versions of code.
SVN -> sub version control

now GIT come with distributed control system , before pushing the code to GIT , we save our code to VSC (this is called as workspace) , after development completes we will add those files to staging area , then we will commit to local repo and then we will push that code to GIT.
vsc (workspace) --> temp area(add) --> local repo(commit) --  git (remote) (push)
when the code increases or new versions came, GIT only tracks difference of code and saves that only. so memory doesn't gets wasted. in every system where git is connected , we can see all versions of code.
when system crashes , with the help of one new system to restore the entire history
