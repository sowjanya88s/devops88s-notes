.tf files -- desired/declared/expected/asked infra

here declarative means it does not follows the order and whatever u write u will get it and it works based on dependancies and u should follow the syntax

actual infra -- what is actually created inside provider

remote sate
===========
so in colloboration env, state should be stored in remote like s3 so that we can prevent errors and duplicates , it should be locked , parallel modifications should not be allowed
if a person work is actively creating , updating and destroying the state file should be locked

now lets use remote state
in infra parallel changes are not allowed
because if one process is running , modifications are under progress , now allowing 2nd person to make changes in the same state file --  there will be lot of confusion -- so terraform blocks modifying state files by multiple persons at one time

provider.tf
===========
terraform {
  required_providers {
    aws = {
      source = "hashicorp/aws"
      version = "6.33.0" # Terraform AWS provider version
    }
  }

  backend "s3" {
    bucket  = "remote-state-aws-88s" # Replace with your unique bucket name
    key     = "remote-state.tfstate"  # name of the state file
    region  = "us-east-1"
    encrypt = true
    use_lockfile   = true  # to lock state file while applying infra
  }
}

provider "aws" {
  region = "us-east-1"
}

provisioners
============
there are two types of provisioners
1. local-exec -- where terraform executes (for ex: now we r running terraform in windows , so windows is local because we are executing terraform commands in windows only)
2. remote-exec -- executes inside the resources created by terraform

if we want to take some actions or to run some scripts we can use terraform

provisioners will be executed at the time of destroy or creation but not at the time of updating  resources

if we don't specified when to run provisioners -- they will run at the time of creation

if we specified to run provisioners at the time of destroy -- they will run before destroying resources 

so u can mention multiple provisioners at the time of creation or destroy resources and they will be executed in order

for ex: after creation of ec2 instance, we can write ip of created ec2 instance in inventory file (creates inventory.ini at that time)
in provisioner we can use self variable , self is the special variable
after terraform apply -- u can see ip in inventory
by this we can handover to ansible for configuring or we can run scripts or commands
provisioners will not applied at update time
when u throw error by using provisioner -- terraform stops execution there -- terraform apply fails
on_failure = continue -- terraform ignores the errors in provisioners
when error comes -- terraform deletes the previous resources -- creates them once again

lets see remote-exec
by using this we can connect to resources and can run some commands inside it
for that we should connect to instance -- for that we should write connection block -- in taht write protocol name , username , password , public-ip
by using inline block -- commands mentioned in that will be executed one by one
while running remote-exec -- terraform take connection block ans connects to instance -- then executes nginx install ans nginx start commands
we use these provisioners to execute ansible playbooks 

before destroying instance stop the service of nginx to avoid errors by that it does note accepts new requests

ec2.tf
=======
resource "aws_instance" "example" {
  ami           = "ami-0220d79f3f480ecf5"
  instance_type = "t3.micro"
  vpc_security_group_ids = [aws_security_group.allow_tls.id]

  # self is the special variable
  provisioner "local-exec" {
    command = "echo ${self.public_ip} > inventory.ini"
  }

  provisioner "local-exec" {
    command = "exit 1"  #throwing error
    on_failure = continue  #continue after error
  }

  provisioner "local-exec" {
    command = "echo script-2"
  }

  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Deleting the instance'"
  }

  provisioner "local-exec" {
    when    = destroy
    command = "echo > inventory.ini"  # emptying public ip in inventory file
  }

  connection {
    type     = "ssh"
    user     = "ec2-user"
    password = "DevOps321"
    host     = self.public_ip
  }

  provisioner "remote-exec" {
    inline = [
      "sudo dnf install nginx -y",
      "sudo systemctl start nginx"
    ]
  }

  provisioner "remote-exec" {
    inline = [
      "sudo systemctl stop nginx"
    ]
    when = destroy
  }

  tags = {
    Name = "provisioners-demo"
    Project = "roboshop"
  }
}
