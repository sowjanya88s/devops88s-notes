NACL (network access control list)
==================================
consider u send a request in browser , this goes through the internet & gets the response back to ur browser -- consider that ur home network contains multiple devices , how the response comes tour browser only?
why the response does not goes to another system?

when we request something --
that request contains destination port & destination ip
the request also contains source port & source ip -- by this response only comes to our system

https://facebook.com:443

ports available totally -- 0-65,535

when u request something -- a random port will be opened in our system -- i.e called ephemeral port
in that random port a process will be waiting for the response -- once the response is received that process id & port will be closed

for ex:
source ip: 192.168.1.3
source port: 32,897

search -- ephemeral ports range
in old systems , 
0-1023 -> system ports
1024-5000  -> ephemeral range in old systems

search -- ephemeral ports range in aws
in new systems , 
32768-65535 -> ephemeral port range
ephemeral -- temporary

NACL vs SG
===========
SG -> will be attached to instance
   -> source can be IP address or other SG ID
   -> we can't add DENY rule
   -> Rules are stateful, if we are allowing port number 8080 from 10.0.1.34. while sending response back to the source an ephemeral port is opened. since SG is stateful there is no need to write seperate outbound rule
means when we allow one ip (on particular port) in incoming traffic then outbound rule is automatically applied to same ip (on random ephemeral port)
   
NACL -> attached to subnet level
     -> source can be subnet ID
     -> NACL is first layer of defence, then SG
     -> You can add DENY rule..(we can block a ip from here)
     -> NACL are stateless. you have to write outbound rules also...
