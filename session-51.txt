1. Build the image -> Bare min OS + System packages + App run time + Code + dependencies
2. Run the image -> port, name, background, other component urls, etc.

docker run -d

CMD
====
used to run the container. executed at the time of running the container. it will not execute at the time of building..

we can have multiple RUN instructions inside dockerfile. but only one CMD instruction

after installing nginx -- we have to start it, for that

systemctl start nginx -> won't work inside images, systemctl is kernel level process

containers can't access kernels
search -- nginx start command inside container

syntax: ["executable","param1","param2"]

create repo -- CMD --
Dockerfile
==========
FROM redhat/ubi9
RUN dnf install nginx -y
CMD ["nginx","-g",:deamon off;"]

firstly build the image --
cd dockerfiles/CMD
docker build -t cmd:1.0.0 .
by this first 2 commands will be executed
3rd command will be executed while running the image

now run the container in foreground
docker run cmd:1.0.0
then we can't do anything on terminal

your command inside dockerfile should run in foreground, but you can take it into background

if the container is not running u can't access the nginx page

to run container in background --
docker run -d -p 80:80 cmd:1.0.0

docker ps
u can see cmd:1.0.0 here & cmd is executing at the time of running

open browser -- http://<public-ip>:80


LABEL
=====
adds the metadata to the image, used for filteration

create repo -- label --

Dockerfile
==========
#almalinux is exactly same as redhat image, but completely free
FROM almalinux:9
LABEL   COURSE="Docker" \
	TRAINER="siva kumar" \
	DURATION="10HRS" \
	AUTHOR="joindevops"
      
this label is useful when we have have lot of images , we can use this labels for filteration

cd /dockerfiles/label
docker build -t label:1.0.0 .
docker images

filter command usage
docker images --filter "label=COURSE=Docker"
u can see the image with the filter 

to see the option 
docker images --help

u can see tags by using below command --
docker inspect label:1.0.0

can we run label image?
docker ps -a
label:1.0.0 exited
we have to give some command to keep container busy
if we didn't provided command -- it uses /bin/bash (its just a shell terminal) -- it does not keeps container busy

now give command to it --
docker run -d label:1.0.0 sleep 15
docker ps
u can see -- label:1.0.0 sleep 15
we r telling container to goto sleep for 15 sec -- which means we r allocating container to do that task
after sleep 15 sec -- it will exits automatically

nginx -g daemon off -- this keeps the container running for infinite time
to keep container running for infinite time -- we have to give some command


EXPOSE
=====
useful to display the port information of container. it will not really open the port but provides information only to the users

if we run nginx -- by default opens 80
if we mentioned as 8080 -> can't open -- consider it as wrong information

create repo -- expose

Dockerfile
==========
FROM redhat/ubi9
RUN dnf install nginx -y
EXPOSE 80
CMD ["nginx","-g",:deamon off;"]

cd /dockerfiles/expose
docker build -t expose:1.0.0 .
docker run -d -p 8080:80 expose:1.0.0
docker ps 
in output we can see ports of container and host
docker inspect expose:1.0.0
-- in exposed ports 80
we mention this to provide info for user

if we mention EXPOSE 8080 also -- we can see exposed ports as 8080 , but it actually runs 80 port only

CMD vs ENTRYPOINT
=================
We can override the command inside CMD instruction at run time... We can't override ENTRYPOINT command if you try to do so it will go and append.. for best results we can always use both CMD and ENTRYPOINT

CMD can be used to provide the default args/inputs to ENTRYPOINT.. if we want our own args we can always override CMD at runtime

Dockerfile
==========
FROM almalinux:9
CMD ["google.com"]
ENTRYPOINT ["ping"]


push to git & pull to server
docker build -t ENTRYPOINT:1.0.0 .
docker run ENTRYPOINT:1.0.0 
by running in foreground we can see whether it is pinging or not

lets override the command in CMD
docker run ENTRYPOINT:1.0.0 facebook.com


once we login to the container we r getting root user access
by providing root access with container -- it can do anything inside server & by that it can disturb other containers
containers should run in storage server
mandatory thing to consider -- never run container with root access..   
atleast one step before command -- we should create other user & use it

ENV vs ARG
==========
ENV -> provide env variables to container
ARG -> provide variables to image

ARG variables can't be accessed inside container, they are accessible only at the of image building. ENV variables can be accessed in container

ARG can be first instruction in a special case to supply the version to base image

consider at the time of image creation we have mentioned image version as 8 , later we want to change it as 9 
instead of changing the version directly in image -- use ARG command in docker to provide version as ARG
if user provided the almalinux version use it otherwise use version 8


create repo -- ARG

Dockerfile
==========
ARG version
FROM almalinux:${version:-8}
ARG COURSE="Docker" \
	TRAINER="SIVA"
RUN echo "Trainer is: ${TRAINER} , version is: ${version}"
CMD ["sleep", "200"]


push to git & pull to server
docker build -t arg:1.0.0 .
u can see almalinux 8 version runs
u can see echo statement of trainer while building the image
but u can't access version after FROM instruction , its purpose is to supply version to FROM instruction

to provide the version at build time --
docker build -t arg:1.0.0 --build-arg version=9 .

docker run -d arg:1.0.0

docker exec -it <container-id> bash
env
we can't see env variables here & version here
