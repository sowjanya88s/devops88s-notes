FROM -> refer base OS
RUN -> to configure image (executed at run time)
CMD -> command executed at the time of container creation (executed at build time)
one image can have multiple RUN instructions & one CMD instruction

LABEL -> metadata of the image
EXPOSE -> port information
ADD -> same as COPY but 2 extra capabilities. can download content directly from internet and untar the file
COPY -> copy the files from local to image
USER -> to mention the user that container should be running on
WORKDIR -> working directory of the image/container
ENV -> env variables to the container
ARG -> variables at build time, cant be accessed inside container. ARG can be first instruction before 	FROM to supply the version
ENTRYPOINT -> command to be executed while creating container. it cant be overridden, if we try it will go append. CMD can be overridden. CMD can supply default args for ENTRYPOINT. we can always override default args

ONBUILD
=======
our image can be used by someone else. but we want to put some conditions to use our image. that can be enforced using ONBUILD instruction..

our nginx image expects index.html while it is used by others.

create repo -- ONBUILD

Dockerfile
===========
From alamalinux:9
RUN yum install nginx -y
RUN rm -rf /usr/share/nginx/html/index.html
#this ONBUILD instruction runs when someone uses this image
ONBUILD COPY index.html /usr/share/nginx/html/index.html
CMD ["nginx", "-g" , "daemon off;"]

networking
==========
to see the ip info in server -- ifconfig

in our home network -- external ip & internal ip will be provided by modem or default gateway
we r connecting to internet by using wifi interface

imagine server is in AWS , AWS is providing internet by using ethernet , with the ethernet (its nothing but server private ip address) only network comes

when we install docker in server  -- it creates docker-0 virtual interface -- this allocates ip addresses to containers -- mongodb & catalogue

docker inspect mongodb 
in out put we can see gateway -- 172.17.0.1
IP address (mongodb) -- 172.17.0.2

to see the network drivers in server --
docker network ls
there will be 2 types of network drivers -- bridge & host
docker bydefault uses bridge network

host network  -- using ethernet directly
bridge network means -- creation of docker interface & allocating ip addresses directly from that interface

when u want to use host network use below command --
docker run --network host nginx
we can directly access the nginx on host port 80 -- open browser -- http://<public-ip>
can we use host port 80 once again? -- no
docker inspect <container-id>
u cant find ip address here

by using host network the container is not secure & we can't reuse multiple ports
while using host network , we can accees mongodb & catalogue by using host network only

when we use bridge network -- traffic flow -- ethernet or host network (80:80)-- docker -- mongodb , catalogue

docker uses by default bridge network
default bridge network containers can't communicate with each other..

so you should create your own bridge network that why catalogue is not able to communicate with mongodb

command to create our own network --
docker create network roboshop

to see networks available in server --
docker network ls
roboshop bridge local

mongodb & catalogue are using default bridge network -- disconnect them from that --
docker network disconnect bridge mongodb
docker network disconnect bridge catalogue

now connect mongodb & catalogue into roboshop network --
docker network connect roboshop mongodb
docker network connect roboshop catalogue

now check --
ifconfig
in output u can see br-<id> -- is the bridge network we have created now -- check its default gateway -- 172.18.0.1
docker inspect mongodb
ip (mongodb) : 172.18.0.2
docker inspect catalogue
ip (catalogue) : 172.18.0.3

when we use our own bridge network -- containers can communicate with each other on their host names
now check --
docker exec -it catalogue bash
u can see catalogue can connect to mongodb

summary:
========
when you install docker, it creates one bridge network with name also as bridge
by default containers get IP address from this network with name bridge
if you use default bridge network you cant communicate between the containers using names
you should create your own bridge network
