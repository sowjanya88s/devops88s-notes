docker architecture 
====================
1. docker checks image is available or not
2. if available it will run
3. if not available , it will pull from the hub
4. creates container out of it
5. sends the output to user


docker daemon -- where we install docker
docker client == docker command line
docker client connects with docker daemon
docker daemon == docker software running

when we give ,
docker run nginx -- docker daemon checks checks whether nginx is available in local or not
if available it creates container out of it & shows to client
if not available pulls form hub, store it in local , create container & shows to client

create one server -- install docker & extend the volume of it
docker compose up -d
as images are not available in this server , so pulls the images from docker hub & creates containers out of it

just building the images & running those is not enough
we have to implement best practices while building & running the images

volumes
=======
docker containers are ephemeral, if we delete the containers it will delete the data also
if we place databases in container, data should not be deleted -- so we have to use volumes

there are two types of volumes 
1. un-named volumes
2. named volumes

run nginx container -- 
docker run -d -p 80:80 nginx
pulls the image from docker hub & creates the container

now connect to nginx container --
docker exec -it <container-id> bash
cd /usr/share/nginx/html
echo "hi" > hi.html

open browser -- http://<public-ip-of-server>:80
we can see hi there

if we delete the container 
docker rm -f 9c

again we r creating nginx container --
docker run -d -p 80:80 nginx
we can't find the previously created data in this container

docker exec -it <container-id> bash
cd /usr/share/nginx/html
we can't see any data here

if we want to save the data even if the container is deleted -- we have to create volumes
create one directory -- 
mkdir nginx-data
echo "<h1> Hi, from volumes<h1>" > index.html
ls -l

we r telling the container to take data from /home/ec2-user/nginx-data & place that in /usr/share/nginx/html & run the container --
docker run -d -p 80:80 -v /home/ec2-user/nginx-data:/usr/share/nginx/html nginx

open browser -- http://<public-ip-of-server>:80
we can see Hi, from volumes there

if we remove container -- it does not deletes the data -- because we have created the data in the host 

create the container again -- we can see the output
docker run -d -p 80:80 -v /home/ec2-user/nginx-data:/usr/share/nginx/html nginx

this works as HD -- once u save the data , u can mount this data to another container using -v -- these are known as un-named volumes -- these are managed by us

2. use multi-stage builds
=========================
multi stage builds have multiple FROM instructions, basically looks like multiple dockerfiles. one dockerfile is to provide the artifact. we can copy the artifact into runtime image

node artifact = code + node_modules for ex: java = .jar files
application which are built & compiled are called as artifacts

when we give npm install 

when u r installing dependancies there might be cache , dev related tools , etc are installed. they are not required for running into production

we ultimately need node artifact = code + node_modules


search -- docker multi stage build syntax
builder contains code & node_modules
so copy /app from builder to 2nd /app

Dockerfile
==========
FROM node:20.20.2-alpine3.23 AS builder //build
WORKDIR /app
COPY package.json .
COPY *.js .
RUN npm install


FROM node:20.20.2-alpine3.23 //runtime
WORKDIR /app
EXPOSE 8080
COPY --from=builder /app /app 
ENV MONGO="true" \
    MONGO_URL="mongodb://mongodb:27017/catalogue"
CMD ["node", "server.js"]

copy the code in to github & pull to server
docker compose build catalogue
docker images
image size reduced from 378 to 246MB

docker compose up -d
open browser -- check functionality of website

there will be max 2 stages in multi stage Dockerfile
its better to use the same base os image in build & runtime

4. don't run as root
====================

disadvantages of docker
=======================
1. in one server we r running entire roboshop project , which is not safe -- u can't rely on single docker host
2. if u run multiple hosts , there need to be some orchestrator
3. consider catalogue is in oner server & cart is in another server -- container to container communication is difficult if we run in docker
4. if catalogue gets more requests -- if there is load on application , there should be load balancing -- not available in docker
5. auto scaling is not available
6. networking -- if we placed one container in one server -- networking between containers is not possible
7. we cannot store secrets in secret manager in docker
8. if server crashes , volume with data is gone -- volume implementation is not good

to address these disadvantages -- orchestration tools came -- they are docker swam , k8s , mesos etc

in docker swam -- implementation is poor

among these k8s is famous
we have best implementation for auto scaling , load balancing , secrets mgmt , networking etc
k8s is open source tool launched by google
by this EKS(aws) , AKS(azure) , GKE(google) , openshift (redhat) released

we r going to limit this docker for image building & for running the images -- we will use k8s
