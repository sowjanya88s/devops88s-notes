load balancing between pods
===========================
communication between two pods -- at first pod -- service name & service port is configured 
at service -- pod 2 , pod3 , pod4 will be registered this service acts like load balancer

registry -- when a new pod comes , pod contains labels , this pod registers itself in registry of service
by this from next request service sends the traffic to this newly added pod too

pod1 -- service -- pod2 , pod3 , pod4

when pod3 wants to reach to pod1 , pod3 should send request request to service related to pod1

pod3 -- service -- pod1

pod to pod communication happens only through service

nodeport
========
used to expose the pod to outside world. cluster ip is subset of nodeport. every node inside eks cluster opens node port

search -- k8s service nodeport
if we mention type as nodeport -- nodeport will be created

if we don't mention port -- 
cluster ip will be created for nodeport & a new port will be opened 

open this new port in nodes sg
for example consider we have 2 nodes in cluster -- open that port in nodes
select any node -- sg -- edit inbound rules -- custom tcp -- 32141 -- anywhere -- 0.0.0.0/0 -- save
now take the public ip of 2 nodes -- open browser -- http://<public-ip>:32141
u can see nginx welcome page
k8s opens port in all worker nodes & when we send request to anyone of the worker nodes that request will be forwarded to cluster ip service & from that service file request goes to the pod

k8s modifies internal networking rules , when request comes to instance -- that will be forwarded to service -- that will be forwarded to end point

we can mention the port for node port , if we did not mention port in service file , it gives random port number from ephemeral port range
we have to add the ports in inbound rules of sg

Replicaset
==========
-> create multiple replicas of your pod. its responsibility is to always run desired no.of pods
search -- create replica set
in spec -- replicas -- mention the no.of pods u wan to create
in template -- metadata -- mention pod definition
replicaset matches the pods with labels mentioned in it -- considers them as its pods
labels plays important role in k8s
mention multiple labels for uniqueness
we can't mention pod name in replica set
by mentioning kind as replica set , we should mention pod definition in it
here pod name will be --  <replica-set-name>-<random-5-chars>

16-replicaset.yaml
==================
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx
  # these labels are replicaset labels
  labels:
    project: roboshop
    purpose: rs-demo
    tier: frontend
    component: frontend
spec:
  replicas: 2
  selector:
    # using these labels replica set selects the pod
    matchLabels:
      project: roboshop
      purpose: rs-demo
      tier: frontend
      component: frontend
  # This is pod definition, pod is subset of replicaset
  template:
    metadata:
      labels:
        project: roboshop
        purpose: rs-demo
        tier: frontend
        component: frontend
    spec:
      containers:
      - name: nginx
        image: nginx
        #image: nginx:stable-alpine3.23-perl

deployment/release
==================
general meaning of deployment -- you have v1 running, v2 is ready
steps -- 
1. stop the application server or component. systemctl stop catalogue
2. remove the v1 code
3. download v2 code
4. restart catalogue. systemctl restart catalogue

manifest.yaml
=============
apiVersion: v1
kind: ConfigMap
metadata:
  name: catalogue
  namespace: roboshop
  labels:
    project: roboshop
    tier: backend
    component: catalogue
data:
  MONGO: "true"
  MONGO_URL: "mongodb://mongodb:27017/catalogue"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: catalogue
  namespace: roboshop
  # these labels are replicaset labels
  labels:
    project: roboshop
    tier: backend
    component: catalogue
spec:
  replicas: 1
  selector:
    # using these labels replica set selects the pod
    matchLabels:
      project: roboshop
      tier: backend
      component: catalogue
  # This is pod definition, pod is subset of replicaset
  template:
    metadata:
      labels:
        project: roboshop
        tier: backend
        component: catalogue
    spec:
      containers:
      - name: catalogue
        image: joindevops/catalogue:1.4.0
        imagePullPolicy: Always # safe to pull everytime, there may be images with same version rebuilt and pushed
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "100m"
            memory: "128Mi"
        readinessProbe:
          # httpGet:
          #   path: /health
          #   port: 8080
          tcpSocket:
            port: 8080
          failureThreshold: 2
          periodSeconds: 10
          initialDelaySeconds: 15
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          failureThreshold: 2
          periodSeconds: 10
        envFrom:
        - configMapRef:
            name: catalogue
---
apiVersion: v1
kind: Service
metadata:
  name: catalogue
  namespace: roboshop
  labels:
    project: roboshop
    tier: backend
    component: catalogue
spec:
  selector:
    # labels are used as selectors
    project: roboshop
    tier: backend
    component: catalogue
  ports:
  - protocol: TCP
    port: 8080 # servie port
    targetPort: 8080 # container port
