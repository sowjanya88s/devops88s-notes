we have to add resources to the containers
configuring health checks is one of the best practices
these are called as probes -- liveness , readiness , startup probes
liveness -- checking whether the pod is healthy or not
for every 5 sec , check if the pod is not healthy then we have to restart the pods

readiness -- whether ur pod is ready to accept the traffic or not
usually application take some time to start/get ready

startup probe -- usually java apps or high memory apps take some more time to start , protect slow starting containers with startup probes

start up --> readiness --> liveness
for nodejs & python applications readiness is enough

in liveness probe ,  by using tcp socket we can check whether a port is opened or not
1. if port is opened then app is running
2. /health is giving response then app is running

example:
 
livenessProbe:
      exec:
        command:
        - cat
        - /tmp/healthy
      initialDelaySeconds: 5
      periodSeconds: 5

readinessProbe:
      tcpSocket:
        port: 8080
      initialDelaySeconds: 15
      periodSeconds: 10

startupProbe:
  httpGet:
    path: /healthz
    port: liveness-port
  failureThreshold: 3
  periodSeconds: 10

initially wait for 15 sec for starting the application , perform health checks for every 10 sec , if 2 health checks are failed consider the pod is failed , and then restart the pod
once container is ready , we have to check liveness probe , don't mention initial delay here , mention tcp socket here to check port is opened or not
performing health check is mandatory for bet practices

there is one more important parameter in k8s -- i.e image pull policy
node pulls the image & consider some problem came -- then we made some changes in the image & pushed the image to docker hub & then we deleted the catalogue pod & recreated the catalogue -- there will be no changes
because in imagepullpolicy -- supported policies -- if not present -- the image is only pulled if it does not already exist on the node , this is the most common policy for production to save bandwidth & speed up startup
so while rebuilding the image change its version

while doing development -- when some issue comes -- we will resolve those issues & we will push the changes with same version until we get success
for this we have to mention imagepullpoicy as always 
always -- the kubelet always queries the registry to resolve the image name to a digest , if the local image digest matches the one in the registry , it uses the cached version ; otherwise , it pulls the new image

for ex:
imagepullplicy: Always

by this even the image is available , it pulls the image from docker hub
