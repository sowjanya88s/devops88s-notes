lets see payment
dependancies (pip) will be installed at os level , they will be at /usr/bin/python3
if we copy this path previous dependancies also comes , we need only dependancies related to our application
so install pip in another path -- by using prefix command
python needs build and development related libraries -- 
1. install dependancies in different path, so that we can copy into another stage
2. build stage python needs more packages build-base, linux-headers and pcre-dev
3. at run time u need only pcre package

previously image size is 478MB by using below best practices the image size will be decreased to 108MB
now use minimal image while building image -- python:3.9.25-alpine3.22
we need to install required package at build time using apk command , these package also adds some cache so use command no-cache
run pip3.9 at build stage in install directory only 
install  build-base, linux-headers and pcre-dev at build package because pip3.9 will be installed at os level so it expects some packages
install pcre package at run time
while running the image get /install from build stage
create roboshop user place /install at /usr/local (/install will be executed automatically)
while copying files also we can mention username

open roboshop-docker -- payment --
Dockerfile
==========
FROM python:3.9.25-alpine3.22 AS builder
WORKDIR /app
RUN apk add --no-cache build-base linux-headers pcre-dev
COPY requirements.txt .
RUN pip3.9 install --prefix=/install -r requirements.txt

FROM python:3.9.25-alpine3.22
EXPOSE 8080
WORKDIR /app
RUN apk add --no-cache pcre
RUN addgroup -S roboshop && adduser -S roboshop -G roboshop
RUN chown -R roboshop:roboshop /app
COPY --from=builder /install /usr/local
COPY --chown=roboshop:roboshop payment.ini *.py requirements.txt .
USER roboshop
CMD ["uwsgi", "--ini", "payment.ini"]

Dockerfile
==========
FROM nginx:1.24.0-alpine3.17-slim

RUN rm /etc/nginx/nginx.conf /etc/nginx/conf.d/default.conf

RUN mkdir -p /var/cache/nginx/client_temp && \
    mkdir -p /var/cache/nginx/proxy_temp && \
    mkdir -p /var/cache/nginx/fastcgi_temp && \
    mkdir -p /var/cache/nginx/uwsgi_temp && \
    mkdir -p /var/cache/nginx/scgi_temp && \
    chown -R nginx:nginx /var/cache/nginx && \
    chown -R nginx:nginx /etc/nginx/ && \
    chmod -R 755 /etc/nginx/ && \
    chown -R nginx:nginx /var/log/nginx

RUN mkdir -p /etc/nginx/ssl/ && \
    chown -R nginx:nginx /etc/nginx/ssl/ && \
    chmod -R 755 /etc/nginx/ssl/

RUN touch /var/run/nginx.pid && \
    chown -R nginx:nginx /var/run/nginx.pid /run/nginx.pid
COPY nginx.conf /etc/nginx/nginx.conf
COPY static /usr/share/nginx/html/
USER nginx
CMD ["nginx", "-g", "daemon off;"]

nginx.conf is also conf -- we can place this file in configmap
here nginx.conf is key & content inside file is value
we can tell k8s to take conf from file 

to place the files inside configmap --
consider that we have a pod & inside the pod container is running , add nginx.conf as volume & we have to tell nginx to mount the file from volume because file occupies some memory
advantage by this is if content in file is changed then we can make changes in configmap and then we can restart the container , by that there is no need to rebuild the image

mention nginx.conf content at configmap level , create volume at pod definition level , refer it from config map , mount the volume at container level
in volume mount -- treats this /etc/nginx/nginx.conf as directory , so we have to mention subpath then it treats /etc/nginx/nginx.conf as file and mount that path with nginx.conf

apiVersion: v1
kind: ConfigMap
metadata:
  name: frontend
  namespace: roboshop
  labels:
    project: roboshop
    tier: frontend
    component: frontend
data:
  nginx.conf: |
    user nginx;
    worker_processes auto;
    error_log /var/log/nginx/error.log notice;
    pid /run/nginx.pid;

    include /usr/share/nginx/modules/*.conf;

    events {
        worker_connections 1024;
    }

    http {
        log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                          '$status $body_bytes_sent "$http_referer" '
                          '"$http_user_agent" "$http_x_forwarded_for"';

        access_log  /var/log/nginx/access.log  main;

        sendfile            on;
        tcp_nopush          on;
        keepalive_timeout   65;
        types_hash_max_size 4096;

        include             /etc/nginx/mime.types;
        default_type        application/octet-stream;

        include /etc/nginx/conf.d/*.conf;

        server {
            listen       80;
            listen       [::]:80;
            server_name  _;
            root         /usr/share/nginx/html;

            include /etc/nginx/default.d/*.conf;

            error_page 404 /404.html;
            location = /404.html {
            }

            error_page 500 502 503 504 /50x.html;
            location = /50x.html {
            }

            location /images/ {
              expires 5s;
              root   /usr/share/nginx/html;
              try_files $uri /images/placeholder.jpg;
            }
            location /api/catalogue/ { proxy_pass http://catalogue:8080/; }
            location /api/user/ { proxy_pass http://user:8080/; }
            location /api/cart/ { proxy_pass http://cart:8080/; }
            location /api/shipping/ { proxy_pass http://shipping:8080/; }
            location /api/payment/ { proxy_pass http://payment:8080/; }

            location /health {
              stub_status on;
              access_log off;
            }

        }
    }
---

k8s node is temporary
the data related to db's will be stored in pods only , once the pod is killed data will be deleted
placing data anywhere in cluster is temporary

so to run stateful applications , data should be placed outside the cluster , in this case when entire cluster is deleted we can create new cluster and we can attach the data there

2. dynamic provisioning
by creating EBS/EFS we can mount these volumes to cluster
in this process there will be no problem even if we create clusters & pods -- we can create the new cluster & add the data here
volumes are cluster level
