  name: test-pd
spec:
  containers:
  - image: nginx
    name: nginx
    volumeMounts:
    - mountPath: /var/log/nginx
      name: nginx-logs

    name: almalinux
    command: ["sleep", "2000"]

      name: nginx-logs
      readOnly: true
  # This is at the pod level

  - name: nginx-logs
    emptyDir:

hostpath
========
1. hostpath is not secure , we can't allow containers/pods to access host data 
2. but in rare cases , we allow pods/containers to access hostpath in read-only mode (to know what's happening inside worker nodes logs are important -- we need to push those logs of nodes outside of the cluster)
3. daemon set makes sure of a pod replica runs on each and every worker node ( it is useful know what is going inside worker nodes for that we need to create daemon set)
4. daemonset pod replicas access the underlying host information through hostpath volumes , and can ship it to external clusters

while worker nodes are adding to the cluster daemonset make sure a replica runs on new node also
we can't create replicas in daemon set , it just makes sure replica runs on each and every node in the cluster
in pod def , we will mount logs to /var/log & accesses the underlying host path & shifts the logs there
this is an administration task , namespace here is kube-system it is an admin namespace

02-host-path.yaml
=================
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd-elasticsearch
  namespace: kube-system
  labels:
    k8s-app: fluentd-logging
spec:
  selector:
    matchLabels:
      name: fluentd-elasticsearch
  template:
    metadata:
      labels:
        name: fluentd-elasticsearch
    spec:
      tolerations:
      # these tolerations are to have the daemonset runnable on control plane nodes
      # remove them if your control plane nodes should not run pods
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      - key: node-role.kubernetes.io/master
        operator: Exists
        effect: NoSchedule
      containers:
      - name: fluentd-elasticsearch
        image: quay.io/fluentd_elasticsearch/fluentd:v5.0.1
        resources:
          limits:
            memory: 200Mi
          requests:
            cpu: 100m
            memory: 200Mi
        volumeMounts:
        - name: varlog
          mountPath: /var/log
          readOnly: true
      # it may be desirable to set a high priority class to ensure that a DaemonSet Pod
      # preempts running Pods
      # priorityClassName: important
      terminationGracePeriodSeconds: 30
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

search -- access modes in k8s volumes
 
ReadWriteOnce (RWO): The volume can be mounted as read-write by a single node at a time. Multiple Pods on that same node can access the volume if allowed. 

ReadOnlyMany (ROX): The volume can be mounted as read-only by many nodes simultaneously. This is ideal for shared configuration or reference data.
 
ReadWriteMany (RWX): The volume can be mounted as read-write by many nodes at once. This requires storage backends that support concurrent writes, such as NFS or CephFS.
 
ReadWriteOncePod (RWOP): Restricts volume access to exactly one Pod in the entire cluster. Introduced in v1.22, it provides strict single-writer guarantees to prevent data corruption.

search -- reclaim policies 
Delete (Default): When the PVC is deleted, the PV and the underlying storage asset (e.g., AWS EBS, Azure Disk) are automatically removed.

Retain: When the PVC is deleted, the PV remains in a "Released" state, allowing for manual recovery of data.

Recycle (Deprecated): This policy performs a basic rm -rf on the volume, allowing it to be reused by another PVC.

03-ebs-static.yaml
===================
apiVersion: v1
kind: PersistentVolume
metadata:
  name: ebs-static
spec:
  accessModes:
  - ReadWriteOnce
  capacity:
    storage: 2Gi # should be less than or equal to your disk size
  csi:
    driver: ebs.csi.aws.com
    fsType: ext4
    volumeHandle: vol-052516247a4a4e158
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: ebs-static
spec:
  storageClassName: "" # Empty string must be explicitly set otherwise default StorageClass will be set
  volumeName: ebs-static # This is  the PV name
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 2Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: html-data
      mountPath: /usr/share/nginx/html
  nodeSelector:
    topology.kubernetes.io/zone: us-east-1d
  volumes:
  - name: html-data
    persistentVolumeClaim:
      claimName: ebs-static

commands -- 
kubectl apply -f 03-ebs-static.yaml
kubectl get pv
kubectl get pvc

04-ebs-sc.yaml
===============
# SC is admin task
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: roboshop-ebs
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Retain

commands --
kubectl create -f 04-ebs-sc.yaml
kubectl get sc

now roboshop project team uses this storage class to create disks
now roboshop project team creates pvc using pv
while creating pvc -- mention its name as ebs-dynamic & in pod definition we will write pvc & then pvc asks storage class , this storage class creates pv , the pv creates disk by using drives
