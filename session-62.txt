statefulset
===========
used for stateful applications where data storage is important
application hits MySQL database cluster
MYSQL db cluster maintains some replicas for MYSQL server , each replica contains one own disk
generally DB gets more read instructions
for ex catalogue, it gets more read requests, so we maintain read replicas more & less replicas for write operations
when one update or write request came on write replica, this should be updated on remaining replicas in the cluster , this activity should be done by DB admins(Data sync)
to update anything on other replicas , that write replica should know other pod ip's , for this we need headless service


1. pod names are standard in statefulset , to keep the pod identity same( names will be created like mysql-0 , mysql-1, mysql-2 etc) , deployment pod name are random

2. pods in statefulset are created in order (firstly it created mysql-0 and attaches volume to it, and then mysql-1 and attches volume to it , because by creating all simultaneously if anything fails, cluster will be failed)
pods will be deleted in reverse order. in deployment pods creation & deletion is random

3.pod in statefulset have its own disk. pods in deployment have shared disk

4. statefulset should have pv , pvc is mandatory. in deployment pv, pvc is not mandatory

5. statefulset requies headless service to find other nodes in the cluster when data replication is required , deployment does not require headless service

6. statefulset pods will have same identity

a service with cluster ip none is called headless service

autoscaling is two types
1. horizontal scaling
2. vertical scaling

vertical scaling
================
consider that we have a server with this config -- 1GB RAM, 8GB HD, 2 cpu's -- these are not enough
we need to increase the resources
for that we have to stop the server, increase the resources, restart the server
in this process,
1. there will be downtime
2. single point of failure
even if we increase the resources , if there is a problem in server -- all workloads will be stopped

horizontal scaling
==================
u don't increase resources in single server, instead we create replicas
1. no downtime 
2. no single point of failure

k8s introduced horizontal pod auto scalar -- to implement horizontal scaling
in horizontal pod auto scaler -- horizontal pod auto scaler connects to deployment resource , horizontal pod auto scaler checks average cpu utilization of pods connect to deployment, if avg cpu utilization increases then it will create the replicas

conditions for horizontal pod auto scaler
=========================================
1. u must configure resource limits
2. there should be metric server running (during creation of k8s cluster -- metric server will be created -- this is an addon in k8s cluster)
3. 

if avg cpu utilization > 20 -- create replicas , it can create upto 10 replicas

taints and tolerations
=======================
taint means paint
if u taint a node, it means Kubernetes scheduler will not schedule any pod in that node

in which cases we should taint a node?
1. we reserve that node to other special workloads
2. may be we can attach GPU node to the cluster, we reserve that for AI/ML workloads
3. DB may accept connections from specific IP, we only send the pods that connects to DB in tainted node

toleration means excuse

syntax to taint a node --
kubectl taint nodes node1 key1=value1:NoSchedule
in above command NoSchedule is called as effect

effects
========
1. NoSchedule -- no new pods will be scheduled on the tainted node

2. PreferNoSchedule -- PreferNoSchedule is a preference or soft version of NoSchedule ( means try not to schedule the incoming pods , so scheduler may schedule)

3. NoExecute -- don't schedule new pods and evict existing pods
this affects pods that are already running on the node as follows:
pods that do not tolerate the taint are evicted immediately
pods that tolerate the taint without specifying tolerationSeconds in their toleration specification remain bound forever
pods that tolerate the taint with specifying tolerationSeconds remain bound for the specified amount of time. after that time elapses, the node lifecycle controller evicts the pods from the node -- 1:10 mins

syntax to taint a node --
kubectl taint nodes <node-name> <key>=<value>:<effect>
lets taint a node
kubectl get nodes
kubectl taint nodes <node-name> project=roboshop:NoSchedule

02-toleration.yaml
==================
apiVersion: v1
kind: Pod
metadata:
  name: nginx-tolerated
  labels:
    env: test
spec:
  containers:
  - name: nginx
    image: nginx
  tolerations:
  - key: "project"
    operator: "Equal"
    value: "roboshop"
    effect: "NoExecute"
