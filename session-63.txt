taints and tolerations
======================
if u taint the node, k8s will not schedule any pod on to that node
1. project specific hardware 
2. any specific workloads like database accept only from specific ip

in our project if AI/ML workloads are running, we will add GPU's to the k8s cluster , there we don't run normal pods , we will run only AI/ML pods , for that purpose we taint the nodes

how goto into a tainted node?
if we want our pod on to tainted node, then we should use toleration , toleration will not gurantee pod scheduling
1. we need to select the node
2. toleration acts as exception/reservation to that node

there are some strategies here --
NoSchedule -- don't schedule incoming pods
NoExecute -- running pods should be evicted

if u taint a node --
kubectl label nodes <node-name> project=roboshop:NoExecute
running pods in this node will be evicted
then this pod goes to scheduler and asks scheduler to schedule it somewhere else
now check pod status--
kubectl get pods
kubectl create -f 01-pod.yaml
now u can see the pod in pending state

to make this pod schedule, we need to add toleration & node selector
by adding above two , k8s scheduler will place the pod into selected node

in toleration section mention effect as NoExecute because we have used the same strategy while tainting the node

02-toleration.yaml
==================
apiVersion: v1
kind: Pod
metadata:
  name: nginx-tolerated
  labels:
    env: test
spec:
  containers:
  - name: nginx
    image: nginx
  # asking for VIP room
  nodeSelector:
    project: roboshop
  # showing booking confirmation
  tolerations:
  - key: "project"
    operator: "Equal"
    value: "roboshop"
    effect: "NoExecute"

kubectl apply -f 02-toleration.yaml

node affinity
=============
node selector does not have multiple options, just key value pairs. but affinity have more options. we can have operators In, NotIn, Exists, DoesNotExist, Gt, Lt etc

requiredDuringSchedulingIgnoredDuringExecution -> hard rule. labels should match pod can't be scheduled

preferredDuringSchedulingIgnoredDuringExecution -> soft rule. if label selectors are not matched, k8s considers another node to schedule

command to untaint a node --
kubectl taint nodes <node-name> project=roboshop:NoExecute-

topology.kubernetes.io/zone=us-east-1c (we can see this in tags of a node) -- topology means we are telling k8s to consider this zone as border

03-node-affinity.yaml
=====================
apiVersion: v1
kind: Pod
metadata:
  name: nginx-affinity
  labels:
    env: test
spec:
  affinity:
    nodeAffinity:
      # Hard Rule
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - us-east-1d
	    - us-east-1c
   containers:
   - name: nginx
     image: nginx

04-node-antiaffinity.yaml
=========================
apiVersion: v1
kind: Pod
metadata:
  name: nginx-anti-affinity
  labels:
    env: test
spec:
  affinity:
    nodeAffinity:
      # Hard Rule
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: NotIn
            values:
            - us-east-1c
          - key: project
            operator: NotIn
            values:
            - amazon
  containers:
  - name: nginx
    image: nginx
  tolerations:
  - key: "hardware"
    operator: "Equal"
    value: "gpu"
    effect: "NoExecute"
  
kubectl apply -f 04-node-antiaffinity.yaml
kubectl get pods -o wide

pod affinity 
============
pods will be attracted with each other

for example, in below program 1st pod is related to database ,this pod will be created in anyone of the node
2nd pod is related to backend , we want pod-2 to run in the node where pod-1 is executing 

05-pod-affinity.yaml
=====================
apiVersion: v1
kind: Pod
metadata:
  name: pod-1
  labels:
    tier: database
spec:
  containers:
  - name: nginx
    image: nginx
---
apiVersion: v1
kind: Pod
metadata:
  name: pod-2
  labels:
    tier: backend
spec:
  affinity:
    podAffinity:
      # Hard Rule
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: tier
            operator: In
            values:
            - database
        topologyKey: topology.kubernetes.io/zone
  containers:
  - name: nginx
    image: nginx

kubectl apply -f 05-pod-affinity.yaml

antiaffinity
============
pods will not be attracted with each other

for example, in below program 1st pod is related to database ,this pod will be created in anyone of the node
2nd pod is related to backend , we want pod-2 should not run in the node where pod-1 is executing 

for this there is dedicated antiaffinity module

06-pod-antiaffinity.yaml
========================
apiVersion: v1
kind: Pod
metadata:
  name: pod-1
  labels:
    tier: database
spec:
  containers:
  - name: nginx
    image: nginx
---
apiVersion: v1
kind: Pod
metadata:
  name: pod-2
  labels:
    tier: backend
spec:
  affinity:
    podAntiAffinity:
      # Hard Rule
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: tier
            operator: In
            values:
            - database
        topologyKey: topology.kubernetes.io/zone
  containers:
  - name: nginx
    image: nginx

kubectl apply -f 06-pod-antiaffinity.yaml
kubectl get pods -o wide

2 pods will be created in different nodes

consider we have backend and cache
if backend and cache are in same node then communication will be fast and this is called as affinity 
2. if backend and cache are in different nodes then communication will be slow this is called as anti affinity

we want application high availability , so create 3 replicas
both backend & cache pods are there in all 3 replicas then high availability will be achieved
implement backend to backend -- anti affinity
implement cache to cache -- anti affinity
implement cache to backend -- affinity 

if we have 3 nodes , one cache pod will be created in each node
if we have 3 nodes , one application pod will be created in each node

07-use-case.yaml
================
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis-cache
spec:
  selector:
    matchLabels:
      app: store
  replicas: 3
  template:
    metadata:
      labels:
        app: store
