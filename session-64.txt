helm has important structure in that firstly we will see charts
open helm documentation -- topics -- charts -- chart information , chart is nothing but application. chart contains name, api version, version, app version, description -- metadata about the chart
charts means group of manifest files
in chart.yaml -- required fields are apiVersion -- its related to k8s, in this v2 is the new version lets use v2
name -- name of the chart , for example we are writing this chart for nginx so give name as nginx
version -- mention version of chart , for ex we r giving version as 0.0.1
appVersion -- which application we are going to deploy using helm is app , here we r deploying nginx so mention version of nginx here
mention description -- useful when u change the version

templates -- all k8s manifest files here with placeholders
now create template folder in helm -- create deployment.yaml file in it & place deployment code in it
here in value of replicas place template -- {{ .Values.deployment.replicas }}
templatize the change-cause -- in value mention template -- {{ .Values.deployment.changeCause }}
in nginx image version mention template -- {{ .Values.deployment.imageVersion }}


create service.yaml -- place service code here
spec -- type -- mention template -- {{ .Values.service.type }} 

values.yaml -- we can maintain the placeholder actual values here
create values.yaml in helm repo -- mention values of variables there
we can maintain env specific values here in separate files -- values-dev.yaml, values-prod.yaml

now create changeCause field here and write some description here

===================

  changeCause: "installing latest version"
  imageVersion: "latest"

  type: LoadBalancer

templates -- deployment.yaml
=============================
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  # these labels are replicaset labels
  labels:
    project: roboshop
    purpose: rs-demo
    tier: frontend
    component: frontend
  annotations:
    kubernetes.io/change-cause: {{ .Values.deployment.changeCause }}
spec:
  replicas: {{ .Values.deployment.replicas }}
  selector:
    # using these labels replica set selects the pod
    matchLabels:
      project: roboshop
      purpose: rs-demo
      tier: frontend
      component: frontend
  # This is pod definition, pod is subset of replicaset
  template:
    metadata:
      labels:
        project: roboshop
        purpose: rs-demo
        tier: frontend
        component: frontend
    spec:
      containers:
      - name: nginx
        image: nginx:{{ .Values.deployment.imageVersion }}

=========================
apiVersion: v1
kind: Service
metadata:
  name: nginx-headless

  selector:
    # labels are used as selectors
    project: roboshop
    component: frontend
  ports:
  - protocol: TCP

RBAC -- Role Based Access Control
=================================

1. in below example we r giving role access for default namespace , in core api group, for pods we r giving get, watch, list permissions

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""] # "" indicates the core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]


2. in below example we r giving cluster role access for a secret, in an api group, 

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  # "namespace" omitted since ClusterRoles are not namespaced
  name: secret-reader
rules:
- apiGroups: [""]
  #
  # at the HTTP level, the name of the resource for accessing Secret
  # objects is "secrets"
  resources: ["secrets"]
  verbs: ["get", "watch", "list"]

3. role-binding -- binding user with role

apiVersion: rbac.authorization.k8s.io/v1
# This role binding allows "jane" to read pods in the "default" namespace.
# You need to already have a Role named "pod-reader" in that namespace.
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
# You can specify more than one "subject"
- kind: User
  name: jane # "name" is case sensitive
  apiGroup: rbac.authorization.k8s.io
roleRef:
  # "roleRef" specifies the binding to a Role / ClusterRole
  kind: Role #this must be Role or ClusterRole
  name: pod-reader # this must match the name of the Role or ClusterRole you wish to bind to
  apiGroup: rbac.authorization.k8s.io


4. cluster role binding -- 

apiVersion: rbac.authorization.k8s.io/v1
# This cluster role binding allows anyone in the "manager" group to read secrets in any namespace.
kind: ClusterRoleBinding
metadata:
  name: read-secrets-global
subjects:
- kind: Group
  name: manager # Name is case sensitive
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
