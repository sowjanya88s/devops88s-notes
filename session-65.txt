01-role.yaml
============
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: roboshop
  name: roboshop-trainee
# permissions
rules:
- apiGroups: [""] # "" indicates the core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]

02-role-binding.yaml
====================
apiVersion: rbac.authorization.k8s.io/v1
# This role binding allows "jane" to read pods in the "default" namespace.
# You need to already have a Role named "pod-reader" in that namespace.
kind: RoleBinding
metadata:
  name: roboshop-trainee
  namespace: roboshop
subjects:
# You can specify more than one "subject"
- kind: User
  name: suresh # "name" is case sensitive
  apiGroup: rbac.authorization.k8s.io
roleRef:
  # "roleRef" specifies the binding to a Role / ClusterRole
  kind: Role #this must be Role or ClusterRole
  name: roboshop-trainee # this must match the name of the Role or ClusterRole you wish to bind to
  apiGroup: rbac.authorization.k8s.io

  namespace: roboshop
  name: roboshop-admin

  resources: ["*"]
  verbs: ["*"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  # "namespace" omitted since ClusterRoles are not namespaced
  name: roboshop-cluster-reader
rules:
- apiGroups: [""]
  #
  # at the HTTP level, the name of the resource for accessing 
  # objects is "persistentvolumes"
  resources: ["persistentvolumes"]
  verbs: ["get", "watch", "list"]
- apiGroups: ["storage.k8s.io"]
  #
  # at the HTTP level, the name of the resource for accessing   
  # objects is "storageclasses"
  resources: ["storageclasses"]
  verbs: ["get", "watch", "list"]

service account 
===============
just like system user in linux

consider that a pod is running in eks , this pod wants to access the secret in secret manager, how to access -- by placing our credentials in this pod & accessing the secret from those credentials is not safe , because everyone can see those credentials & also when we left organization , the access will lost , so these two are not safe options

here we can use service accounts , these are like system users in linux , we will give access to service accounts , 

we will attach permissions to access secret in secrets manager to iam role , 
pods run with service accounts (used for non humans),
we will match service accounts with IAM role,

when we create namespace in k8s , one service account will be created with name default , command to check --
kubectl get sa -n roboshop

open k9s -- goto roboshop namspace -- describe mongodb -- this pod runs with default service account
this default service account does not contains any access

now we have to do --
now create service account
create IAM role & attach permission
run pod with service account

to map k8s with IAM role we need OIDC (open ID connect) provider 

OIDC -- used to provide authentication protocol between two systems , allowing users to sign in to third-party applications securely without sharing their passwords

search -- eks OIDC
this is one time activity, admins will setup this intially
command --
eksctl utils associate-iam-oidc-provider --cluster roboshop --approve
